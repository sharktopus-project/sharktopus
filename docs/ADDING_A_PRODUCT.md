# Adding a product

A *product* in sharktopus is a (model, format) pair the user can fetch —
e.g. GFS 0.25° pgrb2, GFS 0.5° pgrb2, HRRR wrfprsf, NAM-CONUS. This
guide walks through adding one as a contributor, using **HRRR CONUS
pressure-level (`wrfprsf`)** as the worked example.

> **Assumption:** the mirror hosting your product uses the same two
> moving parts every NCEP product uses — a canonical filename
> (`{model}.t{HH}z.{fmt}.f{FFF}`-ish) and a public HTTPS URL. Custom
> auth (RDA) and sidecar `.idx` files are already supported by the
> shared helpers.

## The five places you touch

```
  1. src/sharktopus/webui/products.py            ← add Product(...) entry
  2. src/sharktopus/webui/data/products/          ← ship the catalog JSON
     └── hrrr_wrfprsf.json
  3. src/sharktopus/sources/{mirror}.py           ← generalize build_url
                                                   (or add a sibling file)
  4. src/sharktopus/batch/registry.py             ← register new source name
                                                   (only if you added a file)
  5. docs/ROADMAP.md + CHANGELOG.md               ← document it
```

You **do not** touch: the orchestrator, the priority filter, the spread
queue, the WebUI forms (beyond step 2), quota tracking, wgrib2
wrappers, or the cloud handlers unless you also want serverless
cropping for this product (see "Optional: cloud-crop" below).

## Step 1 — register the product

Open `src/sharktopus/webui/products.py` and append to `PRODUCTS`:

```python
PRODUCTS: tuple[Product, ...] = (
    Product(
        id="gfs.pgrb2.0p25", ...,
    ),
    Product(
        id="hrrr.wrfprsf",
        label="HRRR CONUS (wrfprsf)",
        model="HRRR",
        code="wrfprsf",
        description="NOAA High-Resolution Rapid Refresh, CONUS 3 km, pressure-level forecast.",
        catalog_file="hrrr_wrfprsf.json",
        default_bbox=(21.14, 52.62, -134.1, -60.9),
        sources=("aws_hrrr", "nomads_hrrr"),
    ),
)
```

Fields:

- `id` — URL- and JSON-safe unique id; convention is `<model>.<format>`.
- `code` — what the *fetch layer* sees in `build_url(..., product=<code>)`.
- `default_bbox` — `(lat_s, lat_n, lon_w, lon_e)` in −180..180 lon.
  Drives the Submit map: regional products get a dashed overlay +
  `maxBounds` clamp; world-extent `(-90, 90, -180, 180)` is treated as
  "global" (no clamp, no overlay). Declare it explicitly even for
  global models — the UI contract is uniform.
- `sources` — allowlist of registered source names. Empty tuple = every
  registered source can serve this product (GFS today). Non-empty
  filters the Submit Sources pool to just those chips when this
  product is selected.
- `catalog_file` — variables/levels JSON; drives the Variables & levels
  picker. The UI reloads it via HTMX on product change.

The Submit form also derives a per-product **earliest date** at
render time: the earliest `EARLIEST` across the product's sources (or
across all registered sources if `sources=()`). Nothing to declare on
`Product` — it falls out of the source modules you wire in step 3.

When the user picks this product in the Submit dropdown, these UI bits
update in sync (all client-side, no page reload):

- Map draws/removes the coverage rectangle and clamps/unclamps panning.
- Date pickers' `min` and year-menu floor jump to the product's earliest.
- Sources chip pool hides sources not in `product.sources`.
- Variables & levels catalog refetches for the new `product_id`.

## Step 2 — ship the catalog

The UI's variable/level picker is populated from a JSON the product
owns. Generate it from a real file of that product:

```bash
# For GFS we bundle scripts/generate_gfs_catalog.py. Copy + adapt it:
cp scripts/generate_gfs_catalog.py scripts/generate_hrrr_catalog.py
# Edit the NOMADS URL (gfs/prod/ → hrrr/prod/), the default --out path,
# and the product= literal; the wgrib2 inventory parsing is identical.

./.venv/bin/python scripts/generate_hrrr_catalog.py --download
# → writes src/sharktopus/webui/data/products/hrrr_wrfprsf.json
```

The JSON schema is:

```jsonc
{
  "version": "2026-04-21",       // ISO date of generation
  "product": "wrfprsf",
  "source": "wgrib2 -s hrrr.t00z.wrfprsf.f00.grib2",
  "note": "Generated by scripts/generate_hrrr_catalog.py",
  "level_groups": [
    {"name": "Near-surface", "levels": ["surface", "2 m above ground", ...]},
    {"name": "Isobaric — troposphere", "levels": ["1000 mb", "975 mb", ...]},
    ...
  ],
  "variables": [
    {"name": "TMP", "desc": "", "unit": "",
     "category": "Thermodynamic",
     "levels": ["surface", "2 m above ground", "1000 mb", ...]},
    ...
  ]
}
```

The bundled JSON ships with the wheel via `pyproject.toml`:

```toml
[tool.hatch.build.targets.wheel]
artifacts = [
    ...
    "src/sharktopus/webui/data/**/*.json",
]
```

The `**/*.json` glob already covers any new products subdirectory —
you don't edit `pyproject.toml`.

### Power-user override

Once shipped, any user can swap the catalog locally without rebuilding
the wheel:

```
~/.cache/sharktopus/products/hrrr_wrfprsf.json
```

`webui.catalog.load_catalog("hrrr.wrfprsf")` checks this path first and
falls back to the bundled file. Useful when NOAA adds a variable mid-cycle.

## Step 3 — make a mirror serve the product

There are two shapes depending on the mirror. **3b (sibling file per
cloud+product) is the preferred path for any new model** — it
guarantees that adding a product cannot regress an existing one in
production. 3a exists only for variants that share both the URL path
*and* the filename contract of an already-served product.

### 3a — same mirror already serves a structurally identical product

`sources/aws.py`, `sources/gcloud.py`, `sources/azure.py`,
`sources/nomads.py`, `sources/rda.py` already accept a `product`
keyword on `build_url()` and `fetch_step()`. For a product hosted on
the *same bucket path structure and filename pattern* (rare outside
the GFS family — e.g. `pgrb2.0p50` alongside `pgrb2.0p25`), you can
just pass the new `product` code and things work — the URL builder
substitutes it into the canonical filename.

### 3b — distinct URL pattern per model (HRRR case) — **preferred**

HRRR lives under `hrrr/prod/hrrr.YYYYMMDD/conus/...`, not
`gfs/prod/gfs.YYYYMMDD/00/atmos/...`. Add one file per mirror:

```
src/sharktopus/sources/aws_hrrr.py
src/sharktopus/sources/nomads_hrrr.py
```

Each file is a near-copy of the corresponding GFS source with three
changes:

1. `BASE_URL` and `build_url()` reflect the product's path convention.
2. `EARLIEST` / `RETENTION_DAYS` updated to the mirror's HRRR coverage.
3. Use `sources.base.format_filename(template, cycle=..., fxx=..., product=...)`
   with the new product's template (e.g.
   `"hrrr.t{cycle}z.wrfprsf{fxx:02d}.grib2"`) instead of
   `canonical_filename()` (which is GFS-shaped).

**Rule of thumb:** if 80% of the code is identical to the GFS source,
factor the shared parts into `sources/base.py` helpers rather than
letting `aws.py` and `aws_hrrr.py` drift. **Do not edit the existing
GFS source modules.** Their URL asserts in
`tests/test_sources_mirrors.py` are the load-bearing check that
production paths stay byte-identical.

### Register the source

If you added new source files, register them with the batch layer so
`fetch_batch(priority=[...])` routes by name:

```python
# src/sharktopus/batch/registry.py
_REGISTRY: dict[str, ModuleType] = {
    "nomads": nomads,
    "nomads_filter": nomads_filter,
    ...
    "aws_hrrr": aws_hrrr,          # ← new
    "nomads_hrrr": nomads_hrrr,    # ← new
}
```

Update `DEFAULT_PRIORITY` in `batch/priority.py` if the new sources
should appear in auto-priority for their products.

## Step 4 — tests

Write tests that look like the existing ones, but mock the HTTP
responses to HRRR-shaped URLs:

```python
# tests/test_aws_hrrr.py
def test_build_url():
    assert aws_hrrr.build_url("20260421", "00", 6) \
        == "https://noaa-hrrr-bdp-pds.s3.amazonaws.com/hrrr.20260421/conus/hrrr.t00z.wrfprsff06.grib2"
```

Run the suite:

```bash
./.venv/bin/python -m pytest -q
```

The orchestrator, priority, and availability tests should still pass
unchanged — those are product-agnostic.

## Step 5 — docs

- `docs/ROADMAP.md`: move the new source from "planned" to "done".
- `CHANGELOG.md`: add an entry under `[Unreleased]`.
- `README.md`: update the sources table.

## Optional — cloud-crop for the new product

If you also want serverless cropping, extend the three handlers:

```
src/sharktopus/deploy/aws/handler.py        # Lambda
src/sharktopus/deploy/gcloud/handler.py     # Cloud Run
src/sharktopus/deploy/azure/handler.py      # Container App
```

Each handler is a tiny HTTP/event worker that takes a payload like
`{date, cycle, fxx, product, bbox, variables, levels}`, fetches the
byte-range from the *public* mirror, crops, returns the bytes.

The operational default is **one handler deployment per (cloud,
product) pair** — the GFS Lambda/Cloud Run/Container App stays
pointed at the GFS public bucket, and HRRR gets its own. That way a
bad deploy on HRRR cannot degrade GFS, and free-tier quotas stay
attributable.

Adding a new product means:

1. Copy the handler to a sibling directory
   (`deploy/aws/handler_hrrr.py`, etc.), change `SOURCE_BUCKET` and
   the key construction to the new product's layout.
2. Extend `ALLOWED_PRODUCTS` at the top of the new handler to list
   exactly the product codes that handler is willing to serve —
   anything else returns HTTP 400. Defence in depth: this prevents a
   malicious payload from turning an arbitrary string into an S3
   key.
3. Redeploy via `sharktopus --setup <cloud>` with the new handler
   as the image.

Quota tracking (`sharktopus.cloud.*_quota`) does **not** need changes —
counters are keyed by provider, not product.

## Checklist

Before opening the PR:

- [ ] `Product(...)` entry in `webui/products.py`.
- [ ] `webui/data/products/<model>_<fmt>.json` committed.
- [ ] At least one mirror source can serve the product (step 3a or 3b).
- [ ] Source registered in `batch/registry.py` if new.
- [ ] Tests added (`tests/test_<source>.py`).
- [ ] `docs/ROADMAP.md`, `CHANGELOG.md`, `README.md` updated.
- [ ] `pytest -q` is green.
- [ ] Live smoke: one real fetch with `sharktopus --date ... --product <code>`
      or via the WebUI with the new product selected.

## Design principles

- **Keep the core agnostic.** If your change forces an `if product ==
  "hrrr": ...` branch into `batch/` or `io/`, stop and factor it into
  a per-product module instead.
- **Default to an override file.** Never burn a release cycle because
  a variable list shifted upstream — users can drop an updated JSON in
  `~/.cache/sharktopus/products/`.
- **Non-breaking.** Adding a product must not change behaviour for any
  existing product. No new required parameters on public APIs.
